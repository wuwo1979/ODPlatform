## 一、项目全景（2 分钟）

「先看一下项目长什么样。」

【展示目录结构 Slide】

「这是一个标准的 Python monorepo。核心代码都在 `apps/platform/src/odp_platform/` 下面。你们可以看到有 cli、common、training、inference、webui 这些模块。」

「技术栈是四个组件：Ultralytics 的 YOLO 做检测引擎，Gradio 做 Web 前端，FastAPI 做可选的后端服务，OpenCV 做图像处理。」

「架构上有四个特点值得记住——」

【逐条出现动画】

「第一，**Monorepo**。All in one repo。跨模块改代码，一次 PR 搞定。如果是多仓库，你需要同时改好几个仓库、发好几个 PR、协调好几个合并时间——很痛苦。」

「第二，**Marker 文件**。项目根目录有个 `.odp-workspace` 文件。代码启动时从当前目录往上找这个文件，自动定位根目录。所以项目你可以随便移动，换个文件夹、换个电脑，不用改任何配置。」

「第三，**注册表模式**。想加一个新的数据格式？写一个新文件，加一行 `@register` 装饰器就行了。不用改任何现有代码。」

「第四，**三源合并**。CLI 参数、YAML 配置文件、代码默认值——三个来源按优先级合并，每个字段都能查出来自哪里。」

## 二、五层架构（核心，3 分钟）

【走向白板或切换到架构图 Slide】

「好，接下来这个部分是最核心的——五层架构。我建议大家把注意力集中过来。」

「整个平台从底向上分成五层：**Common、Config、Core、Service、CLI**。」

【指着架构图从下往上讲】

「最底下是 **Common 层**。这一层就是基础设施——路径探测、日志系统、常量定义。它不依赖任何业务模块，但所有模块都依赖它。」

「往上是 **Config 层**。这是一个独立的配置管理子系统。它做的事情是：把 CLI 参数、YAML 文件、代码默认值三个来源合并成一个配置，还能记录每个字段从哪来的。这保证了实验的可复现性。」

「再往上是 **Core 层**。这一层做的是单一业务逻辑——Detector 做检测、draw_detections 画框、FrameSource 读帧。每个文件只做一件事。」

「然后是 **Service 层**。这一层编排业务流程。比如训练一个模型，Service 层会负责：合并配置→加载数据→启动训练→回调通知后端→归档模型。一个函数完成一个完整用例。」

「最顶层是 **CLI 层**。10 个命令行入口。这一层只做参数解析，不做任何业务逻辑。它把解析好的参数传给 Service 层。」

「大家可能注意到，我讲的顺序是反的——从底往上。为什么？因为**依赖方向是从顶到底，但理解顺序是从底到顶**。先搞清基础设施，再看上层怎么用它。」

【稍停顿】

「说两个诚实的话——」

「第一，WebUI 是直接调 Core 层的，没经过 Service。按理说应该中间加一层，但我们现在五个人，加了反而增加维护成本。这是务实的取舍。」

「第二，training 模块里 Service 和 Core 的代码混在同一个文件里。我们知道应该拆开，但当前规模不值得。以后重构。」

## 五、核心设计模式（3 分钟）

【切回 Slide】

「好，功能看完了。接下来讲代码层面怎么保证扩展性。三个核心模式。」

### 模式一：注册表

「这是最值得讲的一个模式。」

「传统做法是写 if/else——如果格式是 VOC 就调这个函数，如果是 COCO 就调那个函数。每新增一个格式，你就要打开那个文件加一个 elif。」

「注册表的做法是：每个格式写一个单独的文件，加一行 `@register` 装饰器。」

```python
@register("labelme")
def convert_labelme(...): ...
```

「新增格式 = 新建一个文件 + 一行装饰器。不碰任何现有代码。」

「这里还有一个面试常考的对比——」

【竖起两根手指】

「**两种调度方式**：data_pipeline 是互斥分发——按格式选一个转换器来执行，就像点菜选一道主食。data_validation 是聚合执行——所有检查项都会跑一遍，一个 ERROR 不影响其他检查继续，就像体检做全部项目。」

### 模式二：Marker 文件路径探测

「很多时候项目代码里会出现硬编码的绝对路径，比如 `C:/Users/xxx/project/data/`。换台电脑就废了。」

「我们的做法是：项目根目录放一个 `.odp-workspace` 文件，代码启动时从当前位置往上找这个文件。项目放到任何路径都能自动定位。」

```python
@cache
def _find_workspace_root(start):
    for parent in [start, *start.parents]:
        if (parent / ".odp-workspace").exists():
            return parent
```

### 模式三：三源合并

「一个配置项最终取什么值？有三个来源——CLI 参数优先级最高，YAML 配置文件次之，代码默认值最低。」

「每个字段的 `TraceRecord` 会记录它的来源链。你可以用 `odp-config trace` 命令查任何一个字段是从哪来的。」

## 六、工程踩坑（2 分钟）

「最后分享几个真实踩过的坑。这些都是代码里的注释直接写着的。」

【逐条讲，可以配合表情】

**坑一：MSMF 帧率暴跌。** 在 Windows 上用 OpenCV 的 MSMF 后端读摄像头，帧率只有 15fps。查了半天发现是硬件色彩转换滤镜导致的。解决方案：设置环境变量 `OPENCV_VIDEOIO_MSMF_ENABLE_HW_TRANSFORMS=0`。

**坑二：参数设置顺序。** OpenCV 的 `set()` 设分辨率是有顺序要求的——必须先设宽高，再设 FOURCC，最后设 FPS。顺序错了，分辨率不生效。而且设完之后必须 `read()` 一帧触发驱动协商，`get()` 读回来的才是真实值。

**坑三：CSV 列名变化。** Ultralytics 升级版本后，`results.csv` 的列名变了，导致训练曲线画不出来。解决方案是做了一层列名适配映射表。

**坑四：日志重复输出。** 每次 import 模块日志都打两遍。原因是 logger 的 propagate 没关。解决方案：`propagate=False` 加上幂等保护，防止重复添加 handler。

**坑五：中文日志乱码。** Windows 控制台显示问号。解决方案：`sys.stdout.reconfigure(encoding="utf-8")`。

## 附录：常见 QA 应答

**Q: 为什么用 Monorepo 而不用多仓库？**
A: 跨模块变更加一个 PR 就够了。多仓库的话，一次改动要提多个 PR，协调合并顺序很麻烦。

**Q: CLI 命令是怎么注册的？**
A: 在 `pyproject.toml` 的 `[project.scripts]` 段写映射，`pip install -e .` 后 pip 自动创建可执行脚本。

**Q: 后端挂了会怎么样？**
A: 训练不受影响。所有 HTTP 请求都有 3 秒超时 + try/except，后端不可达只记一条 warning，不中断训练。

**Q: CUDA OOM 怎么办？**
A: 降 batch 最有效，其次是降 imgsz，再其次是开 AMP（默认已经开了）。实在不行就 `device="cpu"`。

**Q: 最大的架构亮点和隐患分别是什么？**
A: 亮点是五层架构清晰、注册表扩展性好、不依赖后端。隐患是 training 模块 Service 和 Core 混在一起、WebUI 跳过了 Service 层。
